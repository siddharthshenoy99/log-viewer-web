          Essential Event Viewer errors & quick differentiation
          

            

              Reference for Windows Logs → System, Application (where Driver Frameworks
              appears), and Reliability Monitor. Same caveat as the panel header: patterns are educational,
              not a diagnosis. IDs must match your export: provider and message text matter.
            


            

              
Essential Event Viewer errors to check

              

                
Event ID 4101 (Display)

                

                  Often quotes “Display driver nvlddmkm stopped responding and has successfully recovered.” That is a
                  TDR recovery: the GPU missed a deadline and Windows reset it. Frequently driver bugs or
                  corruption, thermals, unstable overclock, weak power to the card, or failing GPU hardware.
                

                
Event ID 10110 / 10111 (Driver Frameworks)

                

                  User-Mode Driver Framework: live device/driver failures; messages may name nvlddmkm or
                  amdkmdag. Strong signal for a direct display-driver crash path. Typical follow-up: DDU
                  clean install of the vendor driver; test at stock clocks. If you expected “10010”, that is a different
                  Windows event family; confirm the digit in Event Viewer.
                

                
Event ID 41 (Kernel-Power)

                

                  Common text: “The system has rebooted without cleanly shutting down first.” Means an unclean shutdown:
                  sudden power loss, hard reset, or crash. Under gaming load this often raises suspicion for PSU capacity,
                  failing PSU, or loose power cables; pair with ID 6008 and timestamps.
                

                
Event ID 63 (Kernel-Power)

                

                  Another Kernel-Power pattern for abnormal shutdown semantics on some builds; interpret next to 41 and 6008.
                

                
Event ID 17 / 18 / 19 /
                  20 (WHEA-Logger)

                

                  Hardware error reporting (wording varies by ID: correctable vs fatal, etc.). Often discussed with
                  PCIe, GPU, RAM, CPU, or motherboard. Always read the full Message and XML
                  component fields; update chipset/BIOS from the OEM if errors persist.
                

                
Event ID 6008 (EventLog)

                

                  Typical text: previous shutdown was unexpected, usually the companion narrative to an unclean shutdown;
                  correlate with Kernel-Power 41.

More patterns (same log)

              

                IDs 13 / 14: display driver stopped responding. LiveKernelEvent /
                parameter codes such as 0x141, 0x1a8, 0x1b8,
                0x193 (TDR-style failures under load). XID (NVIDIA driver): decode the code
                in NVIDIA’s XID documentation (PCIe vs VRAM vs power classes). Some entries use Event ID 0
                on certain builds; we do not auto-count ID 0 (too generic).

Report Viewer

            

              Portable build 
            

          

          
        

        

          

            
Purpose

            

              Read-only helpers for support workflows: structured MSInfo summaries, quick parsing of BSOD / bugcheck
              text, GPU-Z sensor log charts, analysis helpers for exported Windows Event Viewer logs (including
              .evtx), and DxDiag text summaries (display / system / sound blocks), without sending customer
              data to a remote service from this page. The System Information toolbar also includes a Language Adder
              export to capture untranslated labels/tokens so offline language support can keep improving.
              The MSInfo path
              applies tolerant decoding and repair so a wide range of corrupted or partially damaged System Information
              exports can still be read when the underlying content is recoverable. Summaries and labels also support
              multilingual exports through local normalization and offline phrase mapping, with dedicated handling for
              French, Russian, Japanese, Turkish, Spanish, Portuguese, and Swedish in addition to English.
            

          

          

            
Privacy & data handling

            
    
              
  • All parsing runs in this browser tab; files are read locally from disk.
    • 
              
  • No background upload pipeline, network use is only when you follow an outbound documentation link.
    • 
              
  • Clearing the page or closing the tab drops in-memory buffers (nothing is written back to disk by this tool).
    • 
            

          

          

            
Limitations (for accurate expectations)

            
    
              
  • BSOD analysis is text-based heuristics and links, not a substitute for kernel debugging with the full dump in WinDbg.
    • 
              
  • GPU charts and “soft checks” are interpretive; validate unusual results against raw logs and hardware context.
    • 
              
  • MSInfo repair covers common XML damage patterns; severely truncated exports may still be incomplete.
    • 
              
  • 
                Event Viewer (.evtx) handling is for locally opened exports in this page; results are summarized
                and keyed events may be incomplete on very large logs or unusual providers compared to Event Viewer on a live
                PC.
              
    • 
              
  • 
                DxDiag parsing expects the standard English section layout from Save All Information;
                localized or older layouts may parse partially. Use the source text panel to verify.
              
    •

Deployment

            

              Ships as static assets in one folder (index.html, styles.css, app.js,
              nvidia-logo.png)
              
                
              
              no npm install, no CDN, no remote API for analysis. Suitable for
              air-gapped or locked-down PCs when policy allows opening local HTML. Use a current evergreen browser
              (Chromium, Firefox, or Edge). See PORTABLE-HANDOFF.txt in the same folder for a full manifest.
